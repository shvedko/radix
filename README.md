## Radix
Высокопроизводительное Sparse Radix Tree на языке Go. Оптимизировано для работы с наносекундными задержками, минимальным потреблением памяти и поддержкой многослойных префиксных запросов.
## 🚀 Особенности

* Sparse Design: Использование битмапов (uint256) и компактных слайсов вместо плотных массивов. Память узла сокращена в 30+ раз.
* Zero-Alloc Search: Итератор спроектирован так, чтобы минимизировать аллокации в куче при поиске и обходе.
* Многослойность: Поддержка "вертикальных" связей (слоев) позволяет использовать дерево как многоколоночный индекс.
* Хирургический Delete: Удаление через итератор со встроенным механизмом схлопывания (Merge) узлов для поддержания идеальной плотности дерева.
* Экстремальная скорость:
* Поиск: ~230 ns
    * Обход (Dump): ~450 ns (на 100 элементов)
    * Удаление: ~15 ns (чистая операция)

go get ://github.com

## 📖 Примеры использования## Базовые операции и Dump

package main
import (
"fmt"
"://github.com"
)
func main() {
t := radix.New[int]()

	// Вставка данных (ключ может состоять из нескольких слоев)
	t.Insert(1, false, []byte("Pavlov"), []byte("Ivan"))
	t.Insert(2, false, []byte("Pavlov"), []byte("Igor"))
	t.Insert(3, false, []byte("Petrov"), []byte("Ivan"))

	// Визуализация структуры дерева
	t.Dump(func(key []byte, prefix []byte, level int, end bool, values []int) bool {
		fmt.Printf("%*s %s: %v\n", level*2, "", string(prefix), values)
		return true
	})
}

## Сложный поиск и фильтрация
Итератор поддерживает nil в качестве wildcard для любого слоя.

// Найти всех с фамилией на "P" и именем на "I"
it := t.Search([]byte("P"), []byte("I"))for it.Next() {
fmt.Println("Found:", it.Get())
}
// Найти всех с любым первым полем, но именем "Ivan"
it = t.Search(nil, []byte("Ivan"))for it.Next() {
fmt.Println("Found on second layer:", it.Get())
}

## Удаление через итератор
Позволяет безопасно удалять элементы во время обхода с автоматической оптимизацией (схлопыванием) дерева.

it := t.Search([]byte("Pavlov"))for it.Next() {
vals := it.Get()
if vals[0] == 1 { // Нашли Ивана
it.Remove() // Удалит Ивана и схлопнет узлы, если ветка опустела
}
}

## ⚠️ Важно

1. Владение памятью: Дерево не копирует префиксы при вставке (Zero-copy стратегия). Убедитесь, что исходные данные не изменяются во время жизни дерева.
2. Потокобезопасность: Пакет не является потокобезопасным. Используйте sync.RWMutex или шардирование для конкурентного доступа.

## 🧠 Глубокое погружение: Как это работает
В основе Radix лежат три инженерных решения, которые позволяют обходить ограничения стандартных реализаций.
## 1. Sparse Index & Bitmaps (POPCNT)
Вместо хранения 256 указателей в каждом узле (2 КБ), мы используем Bitmap-индексацию.

* Индекс: Четыре uint64 (256 бит) хранят маску присутствия детей.
* Хранилище: Компактный слайс []*Radix[T], содержащий только живые ветки.
* Rank (POPCNT): Чтобы мгновенно найти адрес ребенка в слайсе, используется операция Rank (подсчет установленных бит до нужной позиции) через инструкцию процессора POPCNT (math/bits.OnesCount64).

// Логический адрес (0..255) -> Физический индекс в слайсе
idx := countBitsBefore(node.index, char)
child := node.children[idx]

## 2. State-Machine Iterator
Поиск и обход реализованы через итеративный стек фреймов ([]frame) и стейт-машину внутри метода Next().

* Zero-Alloc: Итератор переиспользует стек, не создавая новые объекты при спуске вглубь.
* Backtracking: Поддержка многослойных префиксов (через next) позволяет "прыгать" между логическими слоями дерева, сохраняя контекст обхода.
* Cache Locality: Весь фрейм итератора упакован в 16 байт, что позволяет хранить 4 полных уровня дерева в одной кэш-линии L1 (64 байта).

## 3. "Хирургическое" удаление (In-place Merge)
Метод Remove() — это не просто удаление значений. Это алгоритм восходящей оптимизации:

* При удалении последнего ребенка в ветке, узел-родитель самоуничтожается.
* Если у родителя остается ровно один ребенок, происходит схлопывание префиксов (Path Compression).
* Stability: Стейт-машина итератора автоматически адаптируется к мутациям дерева. Если узел, на котором стоял итератор, схлопнулся с ребенком, итератор пересчитывает смещение и продолжает обход без дублей.

## 4. Zero-Copy & Slicing
Пакет использует концепцию "Владения по ссылке":

* Insert не копирует байты, а делает срезы (slices) от оригинальных данных.
* Это позволяет работать со скоростью памяти, но накладывает ответственность на пользователя по управлению временем жизни входных буферов.
