## 🚀 Linked Technical Digest
Linked Arena — это высокопроизводительный аллокатор памяти и формат сериализации данных «in-place», оптимизированный для экстремальных нагрузок (DPI, HFT, Radix-деревья).

## 1. Архитектура памяти

* Segmented Storage: Память разбита на страницы по 128 КБ (16384 гранулы по 8 байт).
* Three-Level Bitset: Иерархическая система битовых карт (`bitset0` -> `bitset1` -> `bitset2`). Позволяет находить свободное место в памяти за константное время (O(1)).
* Zero-Copy Design: Данные пишутся напрямую в гранулы. Чтение и запись используют `unsafe.Slice` для работы на скорости шины памяти.

## 2. Формат упаковки (Encoding Scheme)
Используется префиксное кодирование первой гранулы цепочки для определения типа перехода:

* T.1 (Stream): [0 + 7 бит] — до 128 следующих гранул идут подряд без заголовков. Эффективность метаданных: ~99.3%.
* T.2 (Short Jump): [10 + 6 бит] — прыжок на 1–64 гранулы.
* T.3 (Medium Jump): [110 + тип] — прыжки от 65 до 134 млн гранул (2–4 байта метаданных).
* T.4 (Long Jump): [1110 + 60 бит] — абсолютный 8-байтовый адрес. Позволяет связывать данные в разреженных (sparse) аренах.
* T.5 (Terminal): [11110 + 3 бита] — конец цепочки, последние 0–7 байт данных.

## 3. Производительность (i7-11700F)

* Write: 12.7 ГБ/с (линейная упаковка с разметкой битовых карт).
* Read: 31.0 ГБ/с (потоковый парсинг через курсор).
* Free: 69.6 ГБ/с (мгновенный сброс битов занятости гранул).
* Allocations: 0 B/op (все операции, включая курсоры, выполняются на стеке).

## 🛠 Linked API Digest

## Основные методы (Linked)

| Метод                         | Описание                                                                                                                           |
|-------------------------------|------------------------------------------------------------------------------------------------------------------------------------|
| `write(p []byte) (id uint64)` | Записывает слайс байт, упаковывает его и возвращает `id`. Автоматически аллоцирует новые страницы при необходимости.               |
| `open(id uint64) cursor`      | Создает легковесный курсор для чтения данных, начиная с указанного `id`.                                                           |
| `free(id uint64)`             | Рекурсивно освобождает все гранулы цепочки, начиная с `id`. Обновляет hint для повторного использования места.                     |
| `reset()`                     | Полная очистка арены (сброс всех битовых карт и `hint`). Память не освобождается, а остается готовой для повторного использования. |

## Работа с курсором (Cursor)
Метод `read(p []byte) (n int)` работает по принципу `io.Reader`:

* Streaming: Если `p` большой, вычитывает целые стримы гранул за одну операцию.
* Partial Read: Сохраняет внутреннее состояние (`pid`, `gid`, `off`, `rem`), позволяя дочитывать данные маленькими буферами.
* Return:
  - `n`, количество прочитанных байт.
  - `0`, когда цепочка данных полностью исчерпана.
  - `-1`, в случае ошибки метаданных.

## Внутренние механизмы (Advanced)

* Hint Optimization: `hint` хранит адрес самой «верхней» свободной гранулы, `free()` автоматически подтягивает `hint` вверх, если освобожденное место находится раньше текущего, `write()` смещает `hint` вниз на свободное место.
* Need & Scan: Алгоритм `need(size)` использует инструкции `bits.TrailingZeros64` для мгновенного поиска непрерывных блоков памяти внутри страниц.
* Unmark2: Метод пакетного освобождения битовых карт. Сбрасывает занятость до 64 гранул за одну операцию AND NOT.

Рекомендация по использованию:
