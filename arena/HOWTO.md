------------------------------
## 📦 Linked Arena Technical Digest
Physical Structure:

* Granule: Фиксированный квант 8 байт.
* Page: 128 КБ (16,384 гранулы). Идеально ложится в L2-кэш.
* Hierarchical Bitset: 3-уровневая система поиска свободных мест:
    * L0 (Fullness): 1 бит на страницу (64 страницы в uint64).
    * L1 (bitset256): 4 слова uint64 на страницу (1 бит на группу из 64 гранул).
    * L2 (bitset16k): 256 слов uint64 на страницу (1 бит на гранулу).
* Performance: mark и findFreeInPage работают за ~2.3 нс ($O(1)$).

Adaptive Header Packing (Distance-First):
Заголовок гранулы занимает 1, 2 или 8 байт, упакованных в первые биты.

1. Type 00 (End): 1 байт заголовок [00 + 6 бит длины]. Остаток гранулы — данные.
2. Type 01 (Near): 1 байт заголовок [01 + 6 бит дельты]. Прыжок вперед до +63 гранул в той же странице.
3. Type 10 (Intra-Page): 2 байта заголовок [10 + 14 бит индекса]. Прыжок в любую точку текущей страницы.
4. Type 11 (Jump): 8 байт (целая гранула). Хранит полный RowID следующей гранулы на другой странице. Данных не содержит.

Memory Address (RowID):

* 64-bit Address: [40-50 бит PageID] | [14 бит GranuleID].
* Позволяет адресовать эксабайты памяти при 14-битном индексе внутри страницы.

Public API:

* Write(data []byte) uint64: Нарезает данные на гранулы, линкует их оптимальными заголовками. Начинает поиск с lastPage/lastIdx (циклический аллокатор). 0 B/op (не считая самих страниц).
* Reader(id uint64) Cursor: Создает итератор для чтения.
* Cursor.Read(p []byte) int: Стримит данные из цепочки в буфер p. 0 Allocations. Поддерживает частичное чтение из гранул через c.off.
* Free(id uint64): Рекурсивно проходит цепочку и сбрасывает биты в bitsets. Обновляет хинт аллокатора (lastPage), если освобожденная память «левее» текущего указателя.

Key Optimizations:

* Hot Bit Hint: lastPage/lastIdx предотвращает деградацию поиска $O(N)$ при заполнении арены.
* Signed Delta (Concept): Возможность «хода назад» через отрицательное смещение страниц (в разработке).
* No Size Dependency: Read не требует передачи размера извне — он честно идет до флага 00.

------------------------------
## Что это дает для Tablespace:

1. Linked Arena используется для неиндексированных полей (длинные тексты, логи).
2. Гарантирует бессмертный RowID в Fixed Arena: как бы ни рос текст, адрес первой гранулы не меняется.
3. Максимально плотное хранение: оверхед метаданных в среднем ~2-3%.

## 2. Сложная проверка для Стрима (Переход страницы)
Твоя логика: nP == tmpP + 1 && tmpG == 16383 && nG == 0 — это идеальный стрим. Для Read и Write это физически непрерывный поток данных.
Правильная проверка во Write должна выглядеть так:

for run < 128 && len(p) > (run*8 - 1) {
// Вычисляем, где ДОЛЖНА быть следующая гранула физически
targetP, targetG := tmpP, tmpG + 1
if targetG == pageGranules {
targetP++
targetG = 0
}

    // Проверяем, свободна ли она на самом деле
    nP, nG := a.next(tmpP, tmpG + 1)

    // Если следующая свободная гранула — это именно та, что идет следом в памяти
    if nP == targetP && nG == targetG {
        run++
        tmpP, tmpG = nP, nG
    } else {
        break // Разрыв: либо занято, либо прыжок через дырку
    }
}

## Почему это важно:

1. Для Read: Когда c.stream > 0, ридер просто делает gid++. Если gid дошел до лимита, он делает pid++, gid=0. Поскольку мы при записи гарантировали, что эти гранулы были свободны и шли подряд, ридер прочитает их как один кусок.
2. Для Write: Мы объединяем страницы в один логический "шланг" данных. Это максимально использует пропускную способность шины памяти.

------------------------------
## Итоговый алгоритм "Бесшовного Стрима":

1. Запись: Если next выдает адрес (P, 16383) -> (P+1, 0), мы считаем это одним стримом.
2. Заголовок: Пишем 0 | count.
3. Чтение: Ридер видит count, читает текущую гранулу, делает +1 к координатам (с учетом переноса страницы) и читает следующую без проверки заголовка.
