Это отличная идея, и она дешевле, так как избавляет от ветвлений внутри цикла.
Вы можете создать «маску допустимых позиций» (alignment mask). Если мы ищем блок размера size, нас интересуют только биты 0, size, 2*size.... Мы можем «схлопнуть» информацию о занятости всего блока в один бит, стоящий на его стартовой позиции.
Вот как это работает: если хотя бы один бит в блоке занят, мы помечаем стартовый бит этого блока как «занятый» в нашей временной маске.

func (a *Linked) findInPageAligned(pid uint64, size uint16) (uint16, bool) {
// ... (проверка size > pageGranules) ...

	if size <= 64 {
		for i := 0; i < len(a.bitset2[pid]); i++ {
			mask := a.bitset2[pid][i]
			if mask == ^uint64(0) { continue }

			// 1. Создаем "карту занятых блоков".
			// Для каждого блока размера 'size', если в нем есть хоть одна 1,
			// мы хотим, чтобы в итоговой маске на стартовой позиции блока была 1.
			
			// Это делается "размазыванием" занятых бит вправо внутри каждого блока
			// и последующей фильтрацией только стартовых позиций.
			occupied := mask
			for s := uint16(1); s < size; s <<= 1 {
				occupied |= occupied >> s
			}

			// 2. Оставляем только биты, соответствующие кратным позициям (0, size, 2*size...)
			// Пример для size=4: mask & 0x1111111111111111
			// Формула маски: (111...) / ((1 << size) - 1)
			alignmentMask := uint64(0xFFFFFFFFFFFFFFFF) / ((1 << size) - 1)
			
			// Находим свободную позицию среди валидных (инвертируем occupied)
			freePositions := (^occupied) & alignmentMask

			if freePositions != 0 {
				shift := uint16(bits.TrailingZeros64(freePositions))
				return uint16(i*64) + shift, true
			}
		}
		return 0, false
	}

	// Для size > 64 логика с прыжками по индексам массива остается самой быстрой
	// так как мы оперируем уже целыми uint64 (words).
	// ...
}

## Почему это эффективно:

1. Без циклов: Вместо 64/size итераций вы делаете $\log_2(\text{size})$ быстрых битовых сдвигов (для 8 это всего 3 операции).
2. Параллелизм: Процессор обрабатывает все блоки внутри uint64 одновременно.
3. TrailingZeros64: Вы сразу получаете индекс первого подходящего блока без перебора shift.

## Математический нюанс:
Выражение uint64(0xFFFFFFFFFFFFFFFF) / ((1 << size) - 1) генерирует повторяющуюся маску.

* Для size=2 (010101...)
* Для size=4 (00010001...)
* Для size=8 (00000001...)
* Для size=64 это просто 1.

Вы правы в том, что один цикл заменился на другой, но разница в их природе и стоимости для процессора колоссальна. Вот почему «размазывание» (SWAR — SIMD Within A Register) почти всегда выигрывает у обычного итерирования:
## 1. Предсказуемость и конвейер (Branchless)

* Старый цикл (shift += size): Содержит внутри if (mask & blockMask) == 0. Для процессора это условный переход. Если свободный блок находится в конце uint64 или его вообще нет, процессор будет постоянно ошибаться в предсказании перехода (branch misprediction), что сбрасывает конвейер инструкций.
* Новый цикл (s <<= 1): Количество итераций фиксировано и крайне мало (для size=8 это 3 шага, для size=64 это 6 шагов). В нем нет условий if. Процессор просто пролетает эти инструкции на максимальной скорости.

## 2. Параллелизм на уровне данных
В старом цикле вы проверяете блоки последовательно: «Первый свободен? Нет. Второй? Нет...».
В новом подходе операции occupied |= occupied >> s обрабатывают все биты сразу.

* За 1-й проход (>> 1) мы проверяем пары бит.
* За 2-й проход (>> 2) мы проверяем уже группы по 4 бита.
* За 3-й проход (>> 4) мы проверяем группы по 8 бит.
  Это логарифмическая сложность $O(\log N)$ вместо линейной $O(N)$.

## 3. Эффективность масок
Генерацию alignmentMask (которая через деление) на самом деле можно вынести в таблицу (lookup table) или вычислить один раз, так как size у вас — это ограниченный набор степеней двойки (1, 2, 4, 8, 16, 32, 64).
## Сравнение «в цифрах» для блока size = 8:

* Вариант с shift: До 8 итераций, в каждой: сдвиг, AND, сравнение, условный переход.
* Вариант с размазыванием: 3 итерации (сдвиг + OR), один AND с маской и один TrailingZeros64.
